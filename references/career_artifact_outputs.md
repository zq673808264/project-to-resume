# Career Artifact Outputs

Use this reference when the user wants more than a resume bullet.

## Project Retrospective

Structure:

- Project goal
- Problem solved
- User responsibilities
- Technical stack
- Main modules
- Difficulties and tradeoffs
- Validation method
- Verified results
- Open questions

## Interview Q&A

Generate 6-10 likely questions. Include concise answer outlines for:

- What did you build?
- What was your personal contribution?
- Why did you choose this stack?
- What was the hardest bug or design tradeoff?
- How did you verify correctness?
- What did AI coding tools help with?
- If AI wrote parts of the code, what did you do?
- What would you improve next?

For AI-assisted coding, use this stance:

`AI helped accelerate implementation and exploration, but I was responsible for requirements, architecture choices, code review, debugging, integration, and validation.`

Adjust the wording to match the evidence.

## LinkedIn or Portfolio Copy

Keep it less dense than a resume:

- One sentence project hook.
- Two to three implementation details.
- One sentence about what the user learned, validated, or improved.
- Avoid unsupported numbers.

## GitHub README Positioning

Improve public project presentation with:

- Clear one-line purpose.
- Feature list based on actual functionality.
- Tech stack.
- Architecture or workflow summary.
- Setup and usage if available.
- Screenshots or demo links only when they exist.

## Technical Blog Outline

Use when a project has a meaningful learning story:

- Background and motivation
- Problem decomposition
- Architecture or workflow
- Key implementation details
- Bugs and tradeoffs
- How AI coding tools were used
- Validation and lessons learned
- Future improvements

## Career Pack

When asked for a complete package, produce:

- Resume project entry
- Short version for one-page resume
- Technical version for engineering roles
- English version
- Portfolio introduction
- LinkedIn post
- Interview Q&A
- Blog outline
